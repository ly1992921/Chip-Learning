## 第1节：先给结论

### 1.1 Eyeriss的三大核心结论

**结论一：数据移动是能耗大头，而非计算**

Eyeriss团队通过流片实测发现，在传统CNN加速器架构中，**数据搬运（DRAM/GLB/NoC读写）占据了总能耗的45%~65%，而实际的MAC算术运算仅占不到10%**（[Eyeriss JSSC论文芯片测量结果](https://ieeexplore.ieee.org/document/7738524)）。这一根本性洞察颠覆了"算力 ≈ 性能"的传统思维——你以为什么都在算，其实都在搬数据。

以AlexNet CONV5层为例的功耗分解：

| 组件 | 功耗占比 | 说明 |
|------|---------|------|
| ALU运算（MAC） | <10% | 实际需要的计算只占一小部分 |
| 数据搬运（Spad + NoC + GLB + DRAM） | ~45% | 数据移动才是真正的"能耗杀手" |
| 时钟树 / 控制逻辑 / 静态泄漏 | ~45% | 65nm工艺的固有开销 |

**产品经理视角**：这相当于你开了一家工厂，以为"机器加工"是大头，结果发现"原材料运输"的费用比加工费贵了好几倍。优化运输路线比升级加工设备更立竿见影。

**结论二：Row Stationary数据流比传统方案能效高1.4~2.5倍**

Eyeriss团队将自己提出的RS数据流与三种经典数据流进行了系统对比（[ISCA 2016论文](https://dl.acm.org/doi/10.1145/3007787.3001177)）：

| 数据流类型 | 核心特征 | 相对RS的能效（归一化） | 主要瓶颈 |
|-----------|---------|---------------------|---------|
| **RS (Row Stationary)** ✦ | 同时优化ifmap/filter/psum三种复用 | **1.0×（基准）** | 控制逻辑较复杂 |
| **WS (Weight Stationary)** | 权重固定在PE内，最大化卷积复用 | **1.4× ~ 2.0× 更耗能** | psum移动频繁 |
| **OS (Output Stationary)** | 部分和固定在PE内累加 | **1.6× ~ 2.5× 更耗能** | filter/ifmap移动成本高 |
| **NLR (No Local Reuse)** | 无本地复用，类似传统脉动阵列 | **2.0× ~ 2.5× 更耗能** | 全部数据都需远距离搬运 |

> **数据解读**：RS不仅比最差的NLR好2~2.5倍，也比当时主流的WS方案好1.4~2倍。这意味着在相同的CNN推理任务下，RS架构可以将系统能耗降低一半甚至更多。

**结论三：利用稀疏性可以进一步大幅节能**

CNN中ReLU激活函数会产生大量零值。Eyeriss的实测数据（[CSDN论文阅读笔记](https://blog.csdn.net/qq_40268672/article/details/108809413)）：
- AlexNet CONV2层：输入中约40%的值为零
- AlexNet CONV5层：输入中高达75%的值为零
- 通过数据门控（Data Gating）跳过零值MAC运算，可降低PE阵列45%的功耗
- 通过RLC压缩将DRAM访问量降低43%~62%

### 1.2 芯片实测成绩单

- **AlexNet（Batch Size = 4）**：34.7 fps，278 mW，83.1 GOPS/W，DRAM access/MAC仅为0.0029
- **VGG-16（Batch Size = 3）**：0.7 fps，236 mW，DRAM access/MAC为0.0035

## 第2节：为什么重要

### 2.1 历史背景：2016年AI芯片的"蛮力时代"

在Eyeriss问世之前（2015-2016年），AI加速器的设计思路大体分为两类：

1. **GPU路线**（如NVIDIA Tegra K1）：通用可编程，但能效低。Eyeriss比移动GPU能效高约10倍（[MIT Eyeriss Project](http://eyeriss.mit.edu/)）
2. **ASIC脉动阵列路线**（如谷歌TPU v1，2017年发表）：算力强，但数据复用模式固定，对多样化CNN层适应性差

**这些方案的共同盲区**：缺乏对数据移动能耗的系统性优化。业界主流还在追求"更多MAC、更高频率"，而Eyeriss团队率先用流片数据证明——**节省一次片外访存，比多做一个MAC运算有价值200倍**。

### 2.2 Eyeriss建立了边缘AI芯片的设计方法论

Eyeriss最重要的贡献不是芯片本身，而是**一套完整的方法论**：

1. **数据流分类与分析框架**：首次系统性地将CNN数据流分为WS/OS/NLR/RS四类，并给出量化比较框架。这个框架至今仍是学术界分析AI加速器数据流的标准工具
2. **能量感知设计思路**：将存储层次的能量成本（DRAM > GLB > NoC > Spad）作为首要设计约束
3. **联合优化数据复用**：同时考虑ifmap/filter/psum三种数据类型的复用，而不是只优化一种而牺牲其他
4. **硬件-算法协同设计**：利用CNN的稀疏性（零值跳过）和可压缩性（RLC编码），实现"硬件为算法特性定制"

### 2.3 学术影响力与产业影响

截至2025年，Eyeriss的两篇核心论文（ISCA 2016 + JSSC 2017）已被引用**超过2,500次**，是AI芯片架构领域引用最高的论文之一。它开创的Row Stationary数据流已成为计算机体系结构课程的必修内容。

**受Eyeriss启发的后续设计和产品**：
- 寒武纪DianNao系列 — 同样强调数据流优化
- 华为昇腾310/910 — 继承了数据复用思想
- ARM Ethos系列NPU — 边缘AI推理的设计参考
- 地平线征程系列 — 强调能效优先的边缘AI芯片
- 各类RISC-V AI扩展指令集的设计思路

> **产品经理视角**：如果你需要评估一款边缘AI芯片的"能效真实性"，Eyeriss建立的方法论可以帮你问出关键问题——"你们的芯片数据流是怎么设计的？如何处理数据复用？"

## 第3节：用产品经理背景理解

### 3.1 核心类比：数据流优化 = 仓库物料的搬运路径优化

想象你在管理一个大型电商仓库：

| 仓库概念 | Eyeriss对应 | 能效启示 |
|---------|-------------|---------|
| 中央仓库（DRAM） | 片外DRAM | 去中央仓库取一次货，能量是"在工位旁拿"的200倍 |
| 区域中转站（GLB） | 108KB全局缓存 | 比去中央仓库省6倍，但依然昂贵 |
| 传送带（NoC） | 片上互联网络 | 在同一条传送带上传递比绕远路省得多 |
| 工位旁的物料盒（Spad） | PE本地存储 | 成本最低，但不重复取用就是浪费 |

**传统加速器的问题**（以WS数据流为例）：每个工位（PE）只固定放一种螺丝刀（权重），做完一道工序后把所有半成品（psum）搬到下一站。每次搬运都要经过区域中转站（GLB），甚至退回中央仓库（DRAM）。这就像让流水线上的工人每次拿零件都跑到仓库去取。

**Eyeriss的RS方案的最佳比喻**：把生产线设计成——权重在工位间水平传递（像传送带），输入数据在工位间对角线传递（像接力赛），累加结果垂直汇总（像流水线上传递半成品）。三种物料同时在最优路径上流动，不需要绕路去中央仓库。

### 3.2 类比扩展：三电系统的能量流管理

如果你有新能源汽车三电系统背景，可以对标理解：

| 电动车能量流 | Eyeriss数据流 | 共同逻辑 |
|-------------|-------------|---------|
| 电池→电机直驱最高效 | PE本地Spad直接计算最高效 | 减少中间转换环节 |
| 能量回收（刹车回收能量） | 数据复用（psum本地累加） | 不让能量/数据白白流失 |
| 高压平台降低线路损耗 | RLC数据压缩减少带宽需求 | 提高传输效率 |
| BMS热管理保证电池效率 | 数据门控跳过零值无效计算 | 不做无用功 |

> **产品经理洞察**：Eyeriss本质上是在做一个"数据搬运的能量流管理"——它不追求把发动机（MAC阵列）做得更大，而是优化了从油箱（DRAM）到轮子（PE）整个传动路径的效率。

### 3.3 为什么这对产品经理至关重要？

**当你评估一款边缘AI芯片时，不要只看TOPS算力——那是"发动机排量"。更要看它的数据流架构——那是"传动效率"**。

两个同样标称4 TOPS的NPU，在实际边缘推理中功耗可能相差2~3倍，原因就在于：
- 一个可能70%的能耗浪费在数据搬运上（数据流设计差）
- 另一个可能只有40%能耗在数据搬运上（数据流设计好）

**这对产品的影响**：
- 电池续航：2倍能效差异意味着同样的电池容量，续航差一倍
- 散热成本：更高效的芯片可以用更小的散热器，降低整机成本和体积
- 系统架构：能效更高的芯片允许更简单的供电设计，降低BOM成本

## 第4节：核心问题

### 4.1 CNN推理的三个根本挑战

一个典型的CNN推理任务（以AlexNet为例）：
- 权重总计：~61M参数（244MB存储）
- 乘累加操作：约1.4G MACs
- 各层参数差异巨大：CONV1（11×11核, stride=4）到 CONV5（3×3核, stride=1）

**挑战一：数据量远超片上存储**
AlexNet的244MB权重大大超过当时任何芯片的片上存储（Eyeriss仅108KB GLB + PE Spads）。几乎所有数据都必须存放在片外DRAM中，每次计算都需要频繁地"取→算→存"。

**挑战二：不同Layer的Shape差异大**
一个CNN的不同层，filter size（从11×11到3×3）、channel数、stride都不同。固定数据流的加速器对某些层效率高，对另一些层效率低。传统WS（权重固定）数据流在channel数多时表现好，但在filter size大时效率低。

**挑战三：数据移动能耗 >> 计算能耗**
Eyeriss团队量化了这个差距（[ISCA 2016论文分析框架](https://dl.acm.org/doi/10.1145/3007787.3001177)）：

| 操作 | 相对能耗（归一化到16bit MAC） |
|------|-------------------------|
| 16位MAC运算 | **1×** |
| 读/写PE本地Spad（寄存器/SRAM） | **~1×** |
| 读/写GLB（108KB SRAM） | **~6×** |
| 读/写DRAM（片外） | **~200×** |

> **本质矛盾**：一次DRAM访问的能量 ≈ 200次MAC运算。这意味着——即使你让MAC阵列100%满负荷运行，只要数据搬运路径没优化好，总能耗的大部分就浪费在"路上"，而不是"计算"上。

### 4.2 传统架构为什么低效

**GPU方案**（如NVIDIA Tegra K1）：通用性强，但cache层次的能耗效率远不如专用数据流设计。测量结果显示Eyeriss在AlexNet上的能效（83.1 GOPS/W）比移动GPU高约10倍（[MIT Eyeriss Project](http://eyeriss.mit.edu/)）。

**权重固定（WS）加速器**（如早期脉动阵列方案）：权重在每个PE中保持不动，最大化卷积复用。但psum的移动需要频繁写回GLB，造成高能耗。

**输出固定（OS）加速器**：部分和在PE内累加完成后才输出。但filter和ifmap都需要频繁从GLB读取，在大filter size的层上效率极低。

**根本原因**：所有传统数据流都只优化了某一种数据类型（权重/输入/输出）的复用，而忽视了其他类型的数据移动成本。

## 第5节：总体架构

### 5.1 芯片规格一览

| 参数 | 数值 |
|------|------|
| 工艺节点 | **65nm CMOS**（1.0V / 0.9V） |
| 芯片面积 | **12.6 mm²**（core区域） |
| PE阵列 | **12行 × 14列 = 168个PE** |
| 片上GLB | **108KB SRAM**（100KB ifmap/psum + 8KB filter） |
| 各PE本地Spad | 每个PE约200~400B寄存器/SRAM |
| 工作频率 | **200~250 MHz**（core clock） |
| 片外带宽 | 64-bit DDR @ Link Clock |
| 典型功耗 | **278 mW**（AlexNet, BS=4）/ **236 mW**（VGG-16, BS=3） |
| 计算性能 | **23.1 GOPS**（AlexNet, 35.7 fps）/ 约0.7 fps（VGG-16） |
| 峰值能效 | **83.1 GOPS/W**（AlexNet） |

数据来源：[Eyeriss Project](http://eyeriss.mit.edu/)，[JSSC论文](https://ieeexplore.ieee.org/document/7738524)

### 5.2 系统级架构概览

Eyeriss采用**双时钟域异步设计**，分为Core Clock Domain和Link Clock Domain，通过FIFO异步通信。

**存储层次（从昂贵到便宜）**：

| 层级 | 容量 | 访问能量（相对MAC） | 说明 |
|------|------|-------------------|------|
| DRAM（片外） | 数GB | ~200× | 仅存不下溢的数据，最小化访问 |
| GLB（片上全局缓存） | 108KB | ~6× | 分区的ifmap/psum和filter存储 |
| NoC（片上网） | 通信路径 | ~2× | PE间直接传输，不走GLB |
| Spad（PE本地存储） | ~200-400B/PE | ~1× | 每种数据类型都有独立Spad |

**控制层次**：
- **Top-Level Control**：管理DRAM↔GLB数据传输、NoC路由配置、RLC编解码、ReLU操作
- **Low-Level Control（PE内）**：每个PE有独立的有限状态机（FSM），数据到达即触发计算，无需全局同步

> **PE独立性是Eyeriss的关键创新之一**：传统脉动阵列要求所有PE在同一时钟周期同步操作；Eyeriss的每个PE可以异步处理自己的数据流，设计更灵活，也更适合处理不同shape的CNN层。

### 5.3 片上网络（NoC）

Eyeriss的NoC由三个子网络构成：

1. **GIN（Global Input Network）**：GLB → PE的全局输入网络，支持单周期多播（1-to-N）。采用Y-Bus + X-Bus两级层次结构，通过可配置的Row ID和Col ID实现精准多播路由
2. **GON（Global Output Network）**：PE → GLB的全局输出网络
3. **LN（Local Network）**：PE垂直邻居间的直接通信通道，用于psum的累加传递

**GIN多播机制的关键**：每个从GLB读出的数据带有(row, col)标签，只有ID匹配的X-Bus和PE才接收该数据。未匹配的PE被时钟门控锁定以节能。不同层的数据流映射可通过重新配置ID实现，无需改变物理连线。

## 第6节：核心方法

### 6.1 Row Stationary数据流（核心创新）

**核心思想**：将卷积运算拆解为一维行卷积原语（1D Convolution Primitive），每个PE负责一个滤波器行与一行输入特征图的一维卷积计算，通过PE阵列的空间排布同时实现ifmap、filter和psum三种数据类型的最大复用。

**一维卷积原语**（以3个权重×5个输入为例）：

Step 0: w₀×x₀ + w₁×x₁ + w₂×x₂ = psum₀
Step 1: w₀×x₁ + w₁×x₂ + w₂×x₃ = psum₁
Step 2: w₀×x₂ + w₁×x₃ + w₂×x₄ = psum₂

PE内同时有三种数据复用：
- **权重复用**：w₀/w₁/w₂在每个step都被重复使用（卷积复用）
- **输入数据局部性**：x₁/x₂在step 0和step 1之间被相邻窗口共享
- **psum本地累加**：结果存储在PE本地Spad中

**二维卷积的PE集映射**（以3×3卷积核为例）：
- Filter行水平复用：同一行PE共享相同的filter权重行
- Ifmap对角线复用：相邻PE共享ifmap边界值
- Psum垂直累加：同一列PE垂直传递psum

**高维卷积（N×C×M）的PE映射**：采用时间交错（time-interleaving）：
1. 不同ifmap复用同一filter（N方向）：时间分片
2. 不同filter复用同一ifmap（M方向）：时间交替发送filter
3. 不同channel的psum累积（C方向）：PE内累加

### 6.2 数据复用策略的量化分析

| 复用类型 | 传统WS | 传统OS | Eyeriss RS |
|---------|--------|--------|-----------|
| filter复用 | 最优（PE内固定） | 差（需重复读取） | 优（水平广播+时间交错） |
| ifmap复用 | 差（需广播或重复读） | 差（需重复读取） | 优（对角线广播+时间交错） |
| psum累积 | 差（需频繁写回GLB） | 最优（PE内锁定） | 优（垂直传递+本地累加） |

**关键洞察**：RS数据流在三种复用类型上都达到了"接近最优"的水平，而传统数据流最多只擅长一种。

### 6.3 稀疏性加速

CNN中ReLU激活函数将所有负值映射为0。Eyeriss实测：
- AlexNet CONV2层：~40%的输入值为零
- AlexNet CONV5层：~75%的输入值为零

**数据门控**：检测到零值时锁定MAC Datapath，**可降低PE阵列45%的功耗**。

**运行长度编码（RLC）**：每对(Run, Level)编码5bit连续零值个数+16bit非零值，每三对打包为64-bit word。**DRAM访问量降低43%~62%**。

### 6.4 可重构性设计

1. **NoC路由可配置**：PE的Row/Col ID可编程配置，不同层的映射通过重新配置ID实现
2. **PE计算模式可配置**：每个PE可配置负责的filter行、ifmap行和累加模式
3. **GLB分区可配置**：ifmap/psum和filter区域的比例可动态调整

> **产品经理视角**：这就像一台"可编程的传送带系统"——物理传送带是固定的（168个PE），但可以重新编程"哪些工位是一组"、"物料怎么传递"，以适应不同产品的生产线需求。

## 第7节：关键结果

### 7.1 能量效率对比：RS vs 其他数据流

| 数据流 | 归一化能量 | 主要能量消耗来源 |
|-------|-----------|----------------|
| **RS (Row Stationary)** | **1.00** | NoC + Spad为主，DRAM极少 |
| WS (Weight Stationary) | **1.4× ~ 2.0×** | psum频繁写回GLB+DRAM |
| OS (Output Stationary) | **1.6× ~ 2.5×** | filter/ifmap重复从DRAM读取 |
| NLR (No Local Reuse) | **2.0× ~ 2.5×** | 所有数据都走GLB/DRAM |

| 功耗 | **278 mW** |
| 计算性能 | **23.1 GOPS** |
| 能效 | **83.1 GOPS/W** |
| DRAM访问/MAC | **0.0029** |

### 7.4 功耗分解（AlexNet CONV5层）

| 组件 | 功耗占比 |
|------|---------|
| ALU运算 | ~8% |
| 数据搬运（Spad+NoC+GLB+DRAM） | ~45% |
| 时钟树/控制/泄漏等 | ~47% |

### 7.5 消融实验（各技术的独立贡献）

| 技术特性 | 对能效的贡献 | 机制 |
|---------|------------|------|
| **RS数据流** | 1.4×~2.5×优于传统方案 | 三种数据同时复用，最小化DRAM/GLB访问 |
| **数据门控（跳零）** | PE功耗降低45% | 检测到零值时锁定MAC |
| **RLC压缩** | DRAM带宽降低37%~57% | 消除零值存储和传输 |
| **多播NoC** | 与RS配合使用 | 实现单周期多播到多个PE |

## 第8节：落地启发

### 8.1 对边缘AI芯片选型的指导

**场景一：低功耗实时视觉处理（智能摄像头/无人机/AR眼镜）**
- 选型重点：关注DRAM access/MAC ratio，而非峰值TOPS
- DRAM access/MAC为0.003的芯片（如Eyeriss），比0.01的芯片在相同TOPS下能耗低约3倍
- 确认芯片是否支持零值跳过

**场景二：多模型灵活部署（AIoT网关/边缘服务器）**
- 可重构NoC是关键。关注"模型切换开销"
- 好的设计应该做到"切换即切换"，不需要等待数据缓存刷新

**场景三：高算力云端推理**
- RS的设计思想（"同时三种数据复用"）在高算力芯片上通过分块策略同样可以受益

### 8.2 脉动阵列 vs RS架构的取舍

| 维度 | 脉动阵列（如TPU） | Row Stationary（如Eyeriss） |
|------|-----------------|---------------------------|
| 算力密度 | 高 | 中等（更多面积用于Spad） |
| 多样化CNN适应性 | 有限 | 高（通过可重构配置） |
| 能效（数据复用） | 中等 | 高（同时三种数据复用） |
| 控制复杂度 | 低（全局同步） | 中等（PE独立FSM） |
| 扩展性 | 高（规则网格） | 受限（复杂NoC规模扩展有时序风险） |
| 适用场景 | 中大规模云端推理 | 边缘/移动端低功耗设备 |

**选型建议**：
- 边缘设备（<2 TOPS）：RS类设计更优（能效敏感，层间差异大）
- 云端设备（>100 TOPS）：脉动阵列更优（规模化和算力密度优先）
- 车载设备（安全+实时）：需综合考虑灵活性+确定性

## 第9节：产品决策启发

### 9.1 理解芯片能效指标的四个层次

当芯片供应商告诉你"我们芯片算力4 TOPS，功耗500 mW"时，追问：

**第一层：峰值TOPS vs 真实TOPS**
- "在ResNet-50上的实际推理帧率是多少？"

**第二层：能耗的真实构成**
- "你们的芯片DRAM access/MAC比是多少？"
- Eyeriss的数据：0.0029（优秀），很多芯片在0.01~0.05之间

**第三层：稀疏性利用能力**
- "支持零值跳过吗？哪些数据类型支持？"

**第四层：模型兼容性**
- "支持从AlexNet到MobileNet到Swin Transformer吗？每层PE利用率多少？"

### 9.2 芯片选型的"能效×算力"矩阵

在"能效-算力"矩阵中：
- **Eyeriss类（RS数据流）**：高能效、中等算力 → 适合边缘设备
- **TPU类（脉动阵列）**：中能效、高算力密度 → 适合云端
- **GPU类**：低能效、高灵活性 → 适合训练和通用场景
- **CPU类**：最低能效、最高灵活性 → 适合控制面

**产品经理的选型决策路径**：
1. 先确定功耗预算（电池/散热约束）
2. 再确定算力需求（目标模型×帧率）
3. 在"能效-算力"矩阵中选择合适的架构类型
4. 在同类芯片中比较DRAM access/MAC、零值跳过能力等细节指标

### 9.3 具体决策框架

| 评估维度 | 你的问题 | 优秀答案示例 |
|---------|---------|------------|
| 数据流架构 | "采用什么数据流？为什么？" | "采用类RS数据流，针对目标模型优化" |
| DRAM效率 | "典型模型上的DRAM access/MAC比？" | "<0.005" |
| 零值跳过 | "支持输入/权重的零值跳过吗？" | "双端跳过，可跳过40~70%的MAC" |
| 模型兼容性 | "支持目标模型的各层吗？利用率？" | ">80%利用率（ResNet-50）" |
| 能效数据 | "真实场景GOPS/W数据？" | ">50 GOPS/W（ResNet-50）" |

## 第10节：记忆钩子

> **"你以为AI芯片在算，其实都在搬——数据搬运才是能耗的真正大头。"**

> **"Row Stationary不是一种硬件，而是一种物流优化——让权重横着传、输入斜着传、结果竖着传，三者都不需要走远路。"**

> **"一次DRAM访问的能耗 ≈ 200次MAC运算。省一次片外访存，比多做200次计算还有价值。"**

> **"看AI芯片不要只问'多少TOPS'，要问'数据移动效率多高'——DRAM access/MAC比这个指标比峰值算力更能预测真实能效。"**

> **"Eyeriss证明了：2016年的学术界就已经掌握了制造边缘AI芯片的方法论，但这套方法论直到今天很多商业芯片还没有真正学会。"**
