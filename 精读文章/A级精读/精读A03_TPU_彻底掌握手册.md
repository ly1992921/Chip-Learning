这篇论文宣告了"通用计算已死，领域专用计算为王"的时代来临。

---

## 第1节：先给结论

### 性能对比（核心数据）

| 对比维度 | TPU vs Haswell CPU | TPU vs K80 GPU |
|---------|-------------------|----------------|
| 推理速度（加权平均） | 快15.3倍 | 快14.1倍 |
| 推理速度（改进后TPU'） | 快30-50倍 | 快30-50倍 |
| 能效比（含主机） | 高17-34倍 | 高14-16倍 |
| 能效比（仅加速器增量） | **高41-83倍** | **高25-29倍** |

### 核心发现

1. **TPU比GPU快15-30倍**：同样NN推理任务，TPU吞吐量远超同期K80 GPU
2. **能效比高30-80倍**：每瓦特性能碾压CPU和GPU
3. **Systolic Array是核心武器**：脉动阵列大幅减少数据搬移
4. **面向推理，不面向训练**：TPU v1不支持训练反向传播
5. **K80推理表现不佳**：延迟限制下吞吐仅比CPU快1.1-1.9倍

## 第3节：用产品经理背景理解

- **CPU** = 全能型工人：什么都能干，但一次只干一件事
- **GPU** = 一队工人：可同时干很多事，但管理成本高
- **TPU** = 自动化流水线：只能生产特定产品，但效率产量碾压

类比新能源汽车：**通用DSP**可处理多种任务但不极致，**专用MCU**只针对一个任务优化但极致高效。

---

## 第4节：核心问题

### 为什么GPU在AI推理上效率不高？

**痛点1：延迟约束限制了批量大小**
- 推理有严格延迟要求（如7ms），只能用batch size 1-4
- K80利用率仅10-20%

**痛点2：GPU硬件资源被浪费**
- 线程调度、缓存、分支预测——推理不需要这些

**痛点3：FP32/FP16精度浪费**
- 推理不需要高精度，INT8足够

---

## 第5节：总体架构

### 关键组件

1. **Systolic Array (MXU)**：256×256=65,536个MAC单元，700MHz→92 TOPS，占面积30%
2. **Unified Buffer**：28 MiB，占面积37%——芯片上最大的模块
3. **Weight DRAM**：8 GiB DDR3
4. **Accumulators**：4 MiB，32-bit
5. **控制单元**：12条CISC指令，占面积**仅2%**（CPU是30%+）

### 核心设计哲学

芯片面积近2/3用于数据通路（存储+计算），控制逻辑仅2%——DSA的精髓。

---

## 第6节：核心方法

### Systolic Array原理

一次SRAM访问能耗≈5-10次MAC计算。所以减少访存比优化计算更重要。

脉动阵列让数据在单元间流动，256×256矩阵乘法访存从O(N³)降到O(N²)。

### INT8量化推理
INT8的MAC单元面积和能耗均减少4倍。

### 架构对比

| 维度 | Haswell CPU | K80 GPU | TPU v1 |
|------|-----------|---------|--------|
| MAC单元数 | 2×256-bit SIMD | 2×2496 CUDA cores | **65,536 MACs** |
| 控制逻辑占比 | ~30% | ~20% | **2%** |
| 峰值INT8算力 | ~2 TOPS | ~4 TOPS | **92 TOPS** |
| 推理吞吐量 | 1x | 1.1-1.9x | **15-30x** |
| 能效比（增量） | 1x | ~2x | **41-83x** |

## 第7节：关键结果

### 性能对比（加权平均）

| 应用 | CPU | K80 GPU | TPU | TPU vs GPU |
|------|-----|---------|-----|-----------|
| MLP0 | 1x | 0.9x | 32x | 35.6x |
| LSTM1 | 1x | 0.3x | 15x | 50.0x |
| CNN1 | 1x | 2.1x | 8x | 3.8x |
| 加权平均 | 1x | **1.9x** | **29.2x** | **14.1x** |

### 功耗与能效
- TPU加速器增量功耗仅**48W**（K80是149W）
- 能效比：TPU vs CPU 41-83x，TPU vs GPU 25-29x

### 利用率分析
- MLP/LSTM：内存受限（对带宽敏感）
- CNN：计算受限（对算力敏感）
- K80推理利用率极低（~10-20%）

---

## 第8节：创新点总结

### 技术创新
1. 大规模Systolic Array（256×256）
2. INT8量化推理
3. 28 MiB Unified Buffer替代缓存
4. CISC指令集（12条指令，2%控制逻辑）
5. 确定性执行模型
6. 消歧义加载

### 方法论创新
1. DSA设计范式
2. Roofline Model用于AI推理
3. 真实负载评估
4. 生产部署验证

### 谬误与陷阱
1. 数据中心推理非常重视响应时间
2. 更高时钟频率对内存受限任务提升不大
3. K80推理利用率远低于理论峰值
4. CNN仅占Google推理负载5%（MLP和LSTM占绝大多数）

## 第9节：局限分析（产品视角）

| 局限 | 产品影响 |
|------|---------|
| 只支持推理 | 不能用于模型训练 |
| 灵活性差 | 算子受限、动态形状不支持 |
| Google生态锁定 | TensorFlow/Google Cloud依赖 |
| 不能通用 | 只适合大规模稳定推理任务 |

### 适合/不适合

| 适合 | 不适合 |
|------|--------|
| 大规模稳定推理 | 训练场景 |
| 延迟敏感在线推理 | 频繁迭代模型 |
| 模型结构成熟 | 高精度推理 |

## 第10节：落地启发

### 选型决策

| 维度 | 选专用芯片 | 选GPU |
|------|-----------|-------|
| 工作负载 | 稳定型 | 波动型 |
| 延迟 | <10ms | >10ms |
| 吞吐 | >10,000 QPS | <10,000 QPS |
| 模型变更 | 固定3个月+ | 频繁变化 |
| 精度 | INT8可接受 | 必须FP16/FP32 |

### 混合策略
- 训练→GPU集群；大规模推理→TPU/ASIC集群
- 小规模推理→GPU/CPU混合；A/B测试→GPU

---

## 第11节：产品决策启发

### 决策框架
1. 确定工作负载特征（稳定vs波动、延迟约束、吞吐量）
2. TCO模型评估（硬件成本+电力+运维+迁移成本）
3. 风险对冲（不All-in单平台、使用ONNX开放格式）

### 6个实战启示
1. 不要高估通用性的价值
2. 推理和训练是不同的
3. 延迟是第一优先级
4. 量化是必须的
5. 编译器是芯片的一部分
6. 控制逻辑越少越好
